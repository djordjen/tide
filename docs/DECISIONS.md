# TIDE Decision Log

This document records the current design direction. Dates indicate when a
decision was first recorded, not when implementation was completed.

## Accepted decisions

| Date | Decision | Rationale |
|---|---|---|
| 2026-07-14 | The project name is **TIDE: Terminal Integrated Data Environment**. | Memorable, terminal-oriented, and not tied to a legacy product name. |
| 2026-07-14 | TIDE is a business-application runtime; Textual is its first UI adapter. | Domain behavior must remain independent from presentation technology. |
| 2026-07-14 | Python, Textual, SQLAlchemy, Alembic, Pydantic, SQLite, PostgreSQL, FastAPI, and pytest form the initial ecosystem. | They are mainstream, productive, and replaceable behind clear adapter contracts. |
| 2026-07-14 | YAML is the preferred human authoring format; JSON may be accepted and exported. | Compact YAML is readable and supports comments; the compiler remains format-independent. |
| 2026-07-14 | Models are split by entity and concern rather than stored in one large file. | Smaller files are easier to review, merge, and explain. |
| 2026-07-14 | All source formats compile into one typed, immutable `ApplicationModel`. | Every adapter must share identical metadata semantics. |
| 2026-07-14 | Co-located applications live under `applications/<name>/`, separate from `src/tide/`. | The framework/application ownership boundary should be obvious, while still allowing a web2py-style multi-application distribution. |
| 2026-07-14 | "Model compiler" means semantic metadata compilation, not Python bytecode or native executable compilation. | Validation, resolution, and normalization are required in development and at production startup; deployable Python remains ordinary source/package content. |
| 2026-07-14 | Runtime metadata precedes source generation. | It avoids regeneration conflicts while contracts are evolving. |
| 2026-07-14 | Generated views work without a designer; customizations are deterministic overlays. | Immediate productivity and safe model evolution are more important than early visual tooling. |
| 2026-07-14 | Application services, security, and `RecordSession` sit between every adapter and SQLAlchemy. | TUI, REST, MCP, reports, and web must not duplicate or bypass business rules. |
| 2026-07-14 | REST exposure is explicit through `expose.rest`; FastAPI supplies the initial adapter. | Exposure is security-sensitive and should generate OpenAPI without generated source files. |
| 2026-07-14 | TIDE has separate developer and runtime MCP use cases. | Project inspection and runtime data access have different risk and permission profiles. |
| 2026-07-14 | Security covers entity, row, field, action, report, navigation, and export permissions. | Hiding a control is not sufficient authorization. |
| 2026-07-14 | Protected fields use an internal sentinel, rendered as localized protected content in forms. | A display placeholder must never become a stored value or break typed fields. |
| 2026-07-14 | Cross-file and cross-module relationships are resolved through qualified entity names. | Modules need collision-free references and a global relationship graph. |
| 2026-07-14 | Computed fields, filters, validation, action conditions, and conditional presentation share a safe typed expression system. | One criteria model can be validated and translated consistently without `eval`. |
| 2026-07-14 | TIDE Studio edits metadata through a headless designer service. | Textual, web, and AI designer clients should share validation, undo/redo, and save behavior. |
| 2026-07-14 | Reports use a declarative banded model and secured application services. | This preserves Clarion-style reporting without creating a data-access back door. |
| 2026-07-14 | Metadata source schema `0.1` is explicit and separate from the application version. | Parsing changes cannot silently reinterpret an existing project. |
| 2026-07-14 | The v0.1 YAML reader uses safe construction, strict true/false booleans, duplicate-key rejection, and no YAML merge keys. | Ambiguous scalars and hidden duplicate properties are unsafe authoring behavior. |
| 2026-07-14 | The initial expression parser uses a validated, allow-listed Python AST subset but never `eval`. | It supplies a small familiar grammar while rejecting method calls, arbitrary nodes, and unknown paths. |
| 2026-07-14 | The first SQLAlchemy persistence adapter will be synchronous. | A single transaction model simplifies `RecordSession`; async action handlers remain possible and the adapter boundary permits revisiting this later. |
| 2026-07-14 | Basic optimistic version checking precedes every remote mutation. | REST and MCP writes must not ship with silent lost-update behavior; richer merge UX can follow later. |
| 2026-07-14 | Schema v0.1 is single-tenant per deployment. | Multi-user support is required, but tenancy needs explicit identity, uniqueness, cache, migration, and policy semantics rather than an optional context field. |
| 2026-07-14 | TIDE is distributed under the MIT License. | The project should be freely usable, modifiable, redistributable, sublicensable, and commercially usable with only preservation of the notice and warranty disclaimer. |
| 2026-07-14 | View resolution records leaf-level property provenance. | Defaults, presets, entity settings, base views, and overlays are only maintainable when the final value can identify its source. |
| 2026-07-14 | Compiler handler checks are static and do not import application modules. | Validation must confirm project-local functions without executing untrusted project code. |
| 2026-07-14 | The first application-service adapter is an in-memory repository. | Security, sessions, validation, actions, and concurrency can stabilize before SQLAlchemy mapping concerns are introduced. |
| 2026-07-14 | Persistence owns concurrency-token increments; domain actions own state transitions. | Central version management makes retries and stale-write handling consistent across all actions and adapters. |
| 2026-07-14 | Idempotency replay reauthorizes and reprojects its target. | Cached transport output could leak fields after permission changes; replay must use current authorization. |
| 2026-07-14 | Decimal fields use deterministic decimal arithmetic end to end: fractional literals preserve their source tokens, evaluation uses a private 38-digit round-half-even context, division and averages return `Decimal`, and record services normalize decimal inputs at the service boundary. | Business quantities such as money must not accumulate binary floating-point error or depend on process-global decimal settings. |
| 2026-07-14 | Field values are validated against their declared type at commit; values that cannot represent the type fail validation instead of being stored. | An invoicing runtime must not silently store a string in a date field; the SQLAlchemy adapter would otherwise disagree with the in-memory contract. |
| 2026-07-14 | Unique fields treat null as absent: multiple null values never conflict. | This matches SQL unique-index semantics, keeping the in-memory contract and the future SQLAlchemy adapter in agreement. |
| 2026-07-14 | Expression comparisons are limited to `==`, `!=`, `<`, `<=`, `>`, `>=`; membership `in` and identity `is` are rejected at compile time with `TIDE308`. | Operators the runtime and the future SQL translator cannot honor must fail at authoring time, not at evaluation. |
| 2026-07-14 | Compilation separates error and warning severities; errors fail compilation while warnings remain attached to the normalized model and CLI output. | Advisory diagnostics must be visible without making every non-fatal condition block application startup. |
| 2026-07-14 | Actions fail closed unless they declare a permission or explicitly set `unrestricted: true`; omission is `TIDE226`. | Accidentally omitting metadata must never grant mutation authority to every principal with record read access. |
| 2026-07-14 | Reference values are normalized against the target primary-key type and checked for existence before commit. | The in-memory contract must reject broken relationships before the SQL adapter adds database foreign-key enforcement. |
| 2026-07-15 | Database ownership is explicit: `managed` permits reviewed TIDE migrations, while `legacy` maps externally owned tables and columns under a hard no-DDL rule. | Existing third-party schemas must be usable without risking an implicit create, alter, drop, or migration operation. |
| 2026-07-15 | The first durable repository uses synchronous SQLAlchemy Core tables generated from the normalized model, not ORM domain classes. Construction never emits DDL; managed creation is explicit and legacy creation is refused. | Services already operate on typed dictionaries and `RecordSession`; Core preserves that boundary and makes schema authority visible. |
| 2026-07-15 | Repository queries receive validated `QuerySpec` values and row-policy criteria; SQL adapters must translate supported root predicates, ordering, and limits or fail closed during preflight. | Persistent security cannot load a root table and silently rely on Python post-filtering, while memory and SQL still need equivalent observable results. |
| 2026-07-15 | Create row policies evaluate finalized candidate values, and update policies are repeated in the persistence mutation predicate. | Authorization checked only when an edit session opens can become stale before commit, particularly on externally owned tables without version columns. |
| 2026-07-15 | Microsoft SQL Server through SQLAlchemy Core and `mssql+pyodbc` is the first multi-user database target; SQLite remains the fast local/test adapter and PostgreSQL follows after the SQL Server contract stabilizes. | SQL Server is the primary database used by the project owner, and early dialect tests prevent SQLite-specific schema, expression, identity, and concurrency assumptions from becoming architectural constraints. |
| 2026-07-15 | List continuation uses deterministic keyset predicates and opaque, expiring, server-side cursor state bound to the secured query and principal; the initial store is process-local behind a replaceable protocol. | Keyset paging avoids offset drift and cost, while random stateful tokens avoid exposing protected boundary values or inventing a cryptographic token format before deployment key management is defined. |
| 2026-07-15 | Collection hydration is driven by a bounded repository load plan that requires source-field and target-entity read access and applies target read-row policies in child and supported aggregate/reference SQL. | Loading every child before projection leaks protected rows into the runtime, while silent truncation makes business data incorrect; explicit policy predicates and fail-closed depth/item limits preserve both security and completeness. |
| 2026-07-15 | The first OpenAPI milestone is an adapter-independent 3.1 preview built from generated Pydantic record/page models and restricted to explicitly exposed list/get routes. | Contract generation can be tested with the secured core before FastAPI hosting exists, while mutation transport and authentication remain separate reviewed milestones. |
| 2026-07-15 | Idempotent actions reserve a durable key immediately before handler execution, and audited actions persist a started-to-terminal lifecycle; failed or interrupted keys require reconciliation rather than automatic re-execution. | The framework cannot prove whether an interrupted handler produced external effects, so availability must not be preferred over duplicate mutation risk. |
| 2026-07-15 | Multi-process continuation uses an explicitly managed SQLAlchemy cursor store with exact typed server state, transactionally bounded TTL/capacity, and only SHA-256 hashes of random bearer tokens at rest; the in-memory store remains the local default. | Restarts and load-balanced processes must share pagination state without exposing usable tokens or weakening the legacy database no-DDL boundary. |
| 2026-07-15 | Textual is an optional `tui` adapter dependency; `tide run --demo` may execute only the selected application's fixed `demo_data.py` provider and loads it into an in-memory repository. | Headless deployments should not require UI packages, while a deliberate demo flag provides a visible, database-safe golden slice without embedding application fixtures in the framework. |
| 2026-07-15 | `tide run` may execute an application's fixed `runtime.py` registration hook, whose only framework contract is `configure_runtime(records, actions)`. | Ordinary Python business behavior remains application-owned and explicit, while Textual, future REST, and MCP adapters invoke the same registered generators and action handlers through secured services. |
| 2026-07-15 | Reference fields use compact selectors by default and may opt into a resolved multi-column lookup view; declarative selection assignments copy secured related values into writable drafts through application services. | Small lists should remain quick, large business references need searchable tables, and copied prices must be editable historical snapshots shared consistently by TUI and future GUI adapters. |
| 2026-07-16 | `tide run --database-env` reads a deployment URL only from a named environment variable; managed schema creation requires the separate `--create-schema` flag. Managed runtimes use database-backed cursor/action state, while legacy runtimes retain process-local framework state without adding objects to the external schema. | Application metadata and CLI output must not carry secrets, startup must never perform implicit DDL, and legacy compatibility must not weaken the hard no-structure-change guarantee. |
| 2026-07-16 | Development fake-data profiles are application-owned Python providers invoked only by an explicit `tide db seed` command against an empty managed database; generated records pass through secured services and real actions. | Faker can supply realistic repeatable fixtures without putting business-specific generation in the framework or allowing normal startup to mutate persistent data. |
| 2026-07-16 | Reference views may declare `allow_create` plus a compiler-validated target form; nested **Save & Select** commits the master record independently and applies it to the preserved parent draft only when runtime create authorization succeeds. | Business data entry often discovers missing master data mid-transaction, while permissions, reusable adapter semantics, and explicit transaction boundaries must remain intact. |
| 2026-07-16 | Decimal precision/scale and completed-value regular-expression masks are authoritative service contracts; renderers may add typed numeric masks and early feedback without owning validation. | TUI, future Qt/web clients, and APIs must accept and reject the same values, while each surface remains free to provide an idiomatic editing control. |
| 2026-07-16 | The first executable report kind is a compiler-validated single-record query that builds an immutable secured `ReportDocument`; Textual previews it, standard Python writes print HTML, and the optional ReportLab adapter writes A4 PDF. | Primary-key retrieval stays indexed and row-policy-aware on SQL Server and legacy databases, protected fields fail closed, and future GUI/web surfaces can reuse report semantics without depending on Textual or a browser-to-PDF process. |
| 2026-07-16 | FastAPI is a transport boundary over TIDE services, not a replacement service layer. Routes are opt-in; mutations use typed writable projections, `PATCH`, strong ETag preconditions, and action idempotency. The fixed-token development identity may bind only to loopback, and roles are assigned at server startup rather than accepted from clients. | Future remote TUI, Qt, web, and MCP clients should share one secured server without database credentials; partial updates cannot overwrite omitted/protected fields, concurrent changes fail visibly, and production network exposure waits for HTTPS plus a reviewed OAuth/OIDC adapter. |
| 2026-07-16 | Remote clients authenticate against a versioned session-capability resource, validate it against their compiled application, and centralize wire types, protected values, cursors, ETags, and errors in a reusable client adapter. Plain HTTP is loopback-only and redirects are not followed. | TUI and future Qt clients need rendering hints without treating them as authority, must fail visibly on dependency/model version drift, and must never leak bearer credentials through an unencrypted origin or redirect. |
| 2026-07-16 | Remote Textual mode keeps presentation metadata and transient `RecordSession` drafts client-side, but routes structured queries, reference assignments, commits, and actions through service-shaped HTTP facades. Unsupported remote reports fail closed. | Renderers need responsive local form state without receiving database access or duplicating authoritative validation/security; unavailable transports must disable features rather than silently bypass the server. |
| 2026-07-16 | Remote reports transport a versioned renderer-neutral `ReportDocument`; authorization, secured data loading, formatting, and protected-field checks occur on the server, while Textual performs preview and optional local HTML/PDF rendering. | Raw records must not move to a client merely to generate a report, while renderer-neutral output preserves future Qt/web reuse and avoids coupling the API to one binary export format. |
| 2026-07-16 | Production bearer authentication uses provider-neutral OIDC discovery and JWKS validation with exact issuer/audience, asymmetric algorithm and token-type allow-lists, required expiry/subject/key ID, and explicit external-to-TIDE role mappings. Development identity stays loopback-only; non-loopback serving requires direct Uvicorn TLS. | TUI, future Qt/web, and automation clients can share a standard identity boundary without database credentials or client-selected roles, while fail-closed validation and encrypted transport precede network exposure. Interactive login/refresh and trusted reverse-proxy configuration remain separate reviewed contracts. |
| 2026-07-16 | Runtime MCP begins with metadata-opted schema/record resources and structured search tools on the stable v1 official SDK, using authenticated stateless Streamable HTTP beside REST. The canonical resource URL drives RFC 9728 discovery and Host/Origin protection; every call reconstructs a server-controlled `Channel.MCP` context and reuses `RecordsService`. | Read-only AI access can be tested before writes without creating a parallel database/security path. Exact protected values and principal-bound cursors stay consistent across REST and MCP, while mutations/actions wait for concurrency, idempotency, and audit transport contracts. |
| 2026-07-16 | Developer MCP is a separate local stdio server. It exposes compiled project resources and accepts discriminated logical operations for deterministic, approval-required application proposals, but has no source path, arbitrary Python, apply, shell, test-execution, database, or write tool. | AI-assisted generation should begin with inspectable TIDE vocabulary and semantic diagnostics, while source materialization waits for isolated compilation/testing, exact diffs, stale-base protection, explicit approval, comment-preserving edits, and audit. Runtime data authority must never imply source authority. |
| 2026-07-16 | The initial materialization stage renders new-application proposals only into a framework-created temporary tree, compiles and statically checks them, and returns exact artifacts/digests/diff bound to proposal, empty-base and candidate fingerprints. At this stage fixed transition templates were AST-parsed but not executed, used the server clock for stamps, and the candidate was deleted before return; no apply tool exists. | AI clients need compiler evidence and a fully reviewable change unit before approval. Confinement, deterministic output and cleanup preserve the no-workspace-write boundary; the following decision separately expands the preview to fixed-template integration checks. |
| 2026-07-16 | Candidate preview now generates conventional views and may execute only reviewed transition/sequence templates after compiler/static success, using fresh in-memory services with bounded fixture depth/counts. It checks authorization denial, CRUD, idempotent actions, secured reports, HTML and optional PDF, runs no external command, opens no configured database, discloses execution flags, and deletes the candidate. | A generated application needs runtime evidence before approval, while the proposal schema's lack of code/module/path inputs makes fixed-template execution materially different from custom-code execution. Temporary/in-memory isolation is not an OS sandbox; custom code remains prohibited and actual destination/stale-base approval is still required before apply. |
| 2026-07-17 | New generated applications may be published only through a separate local approval service and interactive CLI. Approval binds proposal, canonical `applications/<id>` destination, absent-base and candidate fingerprints; apply regenerates, revalidates, locks, stages, byte-checks, recompiles and renames the tree, refuses existing/case-colliding/unsafe targets, and records `.tide-apply.json`. Developer MCP remains no-write. | Preview output alone is not informed consent. A conspicuous human command, stale-base recheck, no-overwrite rule, cleanup, exact receipt and source-control diff provide a narrow auditable path without granting an AI arbitrary file or existing-application edit authority. |
| 2026-07-17 | Existing-application design begins with a no-write headless `DesignerService`: typed commands address semantic documents and bounded model paths, atomic batches form undo units, and round-trip YAML preserves comments, quotes and order while temporary candidates are compiled and diffed. | Textual, Qt, Web and AI clients need one safe editing contract. Separating in-memory design from stale-base-approved atomic save keeps invalid intermediate states recoverable and prevents a preview-capable client from silently changing source files. |
| 2026-07-17 | Existing YAML sources may be changed only through a separate local `DesignerSaveService`: preparation binds canonical project, live base, candidate, artifact and diff fingerprints; interactive save locks, restages, byte-checks, recompiles, rechecks each target, replaces files atomically one at a time with reverse rollback, and writes a `.tide/designer/` receipt. Python, source inventory changes and MCP-side save remain prohibited. | A typed edit proposal is not write authority. Exact human approval plus optimistic concurrency, a narrow YAML-only mutation surface, preserved originals and auditable receipts gives all future Designer clients one persistence contract without giving AI or presentation clients arbitrary filesystem access. |
| 2026-07-17 | Designer saves create a strict transaction lock held by an OS byte lock and an fsynced phase journal before source mutation. Explicit recovery reacquires the lock, validates canonical records and actual file hashes, then permits only evidence-bound reverse rollback or cleanup of a fully receipted candidate; it is resumable and fails closed on ambiguity. | A process can disappear between individually atomic replacements. Journal hints alone cannot prove filesystem state, timeouts cannot prove a writer is dead, and blindly rolling back can undo a completed save. OS ownership plus byte-derived recovery preserves safety across interruptions without granting a recovery command arbitrary repair authority. |
| 2026-07-17 | The first Textual Studio editor applies only type-preserving scalar-leaf changes to its process-local `DesignerService` session and exposes validation, diagnostics, undo/redo, exact diff, schema-derived choice/Boolean controls, searchable previews and optionally syntax-colored YAML. Containers, schema version and semantic identities remain locked; closing discards changes until the approved save service is explicitly connected. | A useful visual milestone can exercise the shared command and source-schema models without inventing presentation-side mutation semantics or weakening source authority. Structural edits and renames need atomic cross-document commands, while experimentation must remain safe before persistence UX is reviewed. |
| 2026-07-17 | Studio expert YAML editing uses a bounded whole-document Designer command that strictly parses one existing source, preserves its semantic identity, recompiles in memory and participates in ordinary undo/redo; the editor has no direct save authority. | Experienced developers need a fast searchable source workflow, but raw text must not bypass compiler validation, cross-document rename requirements, source-size limits, exact diff review, or the separately approved persistence boundary. |
| 2026-07-18 | Textual Studio may persist only a valid dirty candidate through an exact-diff review and complete evidence-bound approval phrase passed to the existing `DesignerSaveService`; success reopens a clean session, while stale bases and save locks expose no-write blockers and recovery-preview guidance. | Closing the visual editing loop should reuse the already reviewed transactional YAML-only boundary rather than give widgets filesystem authority. Fresh sessions prevent undo from silently crossing a saved baseline, and recovery remains a separate explicit operation. |
| 2026-07-18 | The first structural Studio contract exposes compiler-resolved table/lookup columns and form/inline left-right field tracks with provenance; local fields may move only within their current track through bounded sequence moves or atomic slot swaps. | A small structural vocabulary provides immediate field-ordering value and can be shared by Textual, Qt and Web without premature free-form layout semantics. Keeping inherited/generated tracks read-only and routing every move through compile, diff, undo and approved save preserves layered metadata ownership. |
| 2026-07-18 | Structural cross-column changes are same-position swaps limited to local slots in the same YAML group; local field add/remove operates only on existing entity fields and updates inline columns/layout together. | Current layout metadata has no portable empty-cell concept, so unrestricted transfers could silently change other field columns or group ownership. Conservative swaps and atomic membership edits are predictable, undoable, compiler-validated, and do not imply entity/schema/database deletion authority. |
| 2026-07-18 | Studio field additions to form/inline views require an explicit local destination group; group management permits bounded labels, create/rename, one-adjacent-group reorder, and empty-only removal, with collection sections acting as fixed barriers. | Explicit destinations remove selection-dependent ambiguity. Collection-aware barriers and empty-only deletion preserve layout ownership and prevent a group operation from implicitly moving collections or deleting fields before those structures have their own reviewed designer contract. |
| 2026-07-18 | Portable form structure uses optional repeated `tab` labels on layout sections, ordered form actions, and ordered collection actions; Studio edits complete section/collection/action tracks and Textual renders the same compiled contract. | Tabs and button placement must be application semantics reusable by future Qt/Web renderers, not a Studio-only illusion. Compatible inline-view checks, bounded action vocabularies, undo/diff/compiler validation, and the existing approved save boundary keep structural edits safe. |
| 2026-07-18 | Studio role/terminal previews are static compiled-model projections: a synthetic TUI principal uses the shared `SecurityEngine`, while fixed terminal presets evaluate declared/estimated layout constraints without records, database access, or application-code execution. | A preview should reveal permission and sizing mistakes without becoming a second authorization implementation or leaking real data. Record-dependent field/action rules must remain explicitly conditional until a real secured record is evaluated by runtime services. |
| 2026-07-18 | Record deletion is a secured application-service operation and exists over REST only when both `expose.rest.operations` and `permissions.delete` explicitly allow it. Versioned records require the observed version; root row criteria remain in the atomic delete; `restrict`, `cascade`, and `set_null` execute transactionally in both repository adapters, including application-side SQL behavior for legacy no-DDL mappings. | Delete is higher-risk than ordinary editing and must fail closed at capability, authorization, row, concurrency, and reference boundaries. Application-side relationship behavior preserves one metadata contract without requiring TIDE to alter third-party schemas, while existing database constraints remain a final integrity boundary. |
| 2026-07-18 | Textual record deletion is rendered only when the resolved browse action list contains `delete` and the shared security view grants it. `confirm_delete` presents the record display value with cancellation focused by default; confirmation passes the selected record's observed version to `RecordsService`, refreshes on success, and explains stable relationship restrictions. | Destructive UI must reflect developer-authored presentation and server-enforced authority without becoming an alternative deletion path. A safe second gesture, useful conflict text, and identical local/remote service facades make the operation testable while preserving renderer independence. |
| 2026-07-18 | Stale-edit inspection and resolution use shared three-way Original/Current/Draft contracts. Textual requires an explicit Current/Mine choice for every genuine overlap, then copies the completed plan into a fresh `RecordSession`; the dialog never writes storage, and current permissions plus workflow immutability are reevaluated before any draft value is retained. | Conflict semantics must be reusable by future Qt/Web clients and remain subordinate to service enforcement. Explicit choices avoid silent data loss, while a concurrent workflow transition can still make a concurrency-safe value illegal, so every resolved draft must pass current business rules and normal validation. |
| 2026-07-18 | Record action history is exposed only through a bounded `AuditHistoryService` guarded by explicit `permissions.audit`; REST requires record `get` exposure, remote capabilities disclose only availability, and public projections omit payloads, protected values, raw keys, and stored key hashes. | Audit visibility is distinct from record read/write authority. One renderer-neutral service keeps local TUI, remote TUI, FastAPI, and future clients consistent while fail-closed metadata and non-disclosure prevent the diagnostic surface from becoming a data-leak path. |
| 2026-07-18 | Successful root CRUD mutations are audited inside `RecordsService` into a separate TIDE-owned `tide_record_audit` table; fields default to name-only capture and require `audit: values` for bounded before/after values, while read-policy fields are redacted before storage and rechecked at history read time. | Capturing at the shared service boundary prevents renderer bypass, a separate table avoids altering the existing action lifecycle or legacy application tables, and two-layer redaction keeps operational history useful without turning it into an alternate protected-data channel. |
| 2026-07-19 | Runtime MCP mutations are independently metadata-opted and use the same generated writable inputs and services as REST; domain actions remain separately exposed. Versioned calls require an observed version, idempotent actions require a key, mutation results carry exact protected wire values and correlation, and audit history has its own permission. | An AI client may automate useful business work without receiving database credentials or a parallel validation path, while explicit capability, authorization, concurrency, idempotency, protected-value, and audit boundaries keep omission and retries fail-closed. |

## Open decisions

- Compatibility policy and migration tooling for the eventual stable metadata 1.0 contract.
- Interactive browser/desktop login, access-token acquisition, and refresh
  strategy for individual identity providers.
- Pagination, keep-together, grouping, image, and configurable page-geometry
  limits for the expanded report engine.
- Stable identifiers and explicit rename representation for schema evolution.
- Exact representation of protected fields in versioned REST contracts.
- Whether direct many-to-many syntax belongs in the first stable model.
- Composite-key representation and database-generated key/refresh behavior for
  broader legacy database compatibility.

## Deferred decisions

- General-purpose or arbitrary source-code generation beyond reviewed templates.
- Public plugin and application marketplaces.
- Alternative native runtime implementations in Go or Rust.
- General-purpose workflow language.
- Full browser delivery of Textual applications versus a dedicated web
  renderer.
